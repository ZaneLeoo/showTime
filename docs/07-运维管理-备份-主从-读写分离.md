## 1. 知识清单

### DCL：用户与权限

```sql
-- 创建用户
CREATE USER 'username'@'host' IDENTIFIED BY 'password';
-- 修改密码
ALTER USER 'username'@'host' IDENTIFIED BY 'new_password';
-- 删除用户
DROP USER 'username'@'host';

-- 授权
GRANT SELECT, INSERT ON showtime.* TO 'app_user'@'%';
GRANT SELECT ON showtime.v_daily_sales TO 'report_user'@'%';
-- 刷新权限
FLUSH PRIVILEGES;
-- 查看权限
SHOW GRANTS FOR 'username'@'host';
-- 撤销权限
REVOKE DELETE ON showtime.* FROM 'username'@'host';
```

### 备份与恢复

```sql
-- mysqldump：全量备份
mysqldump -u root -p showtime > showtime_bak.sql

-- 只备份结构
mysqldump -u root -p --no-data showtime > showtime_schema.sql

-- 只备份数据
mysqldump -u root -p --no-create-info showtime > showtime_data.sql

-- 恢复
mysql -u root -p showtime < showtime_bak.sql

-- 一致性备份（InnoDB，不锁表）
mysqldump -u root -p --single-transaction showtime > showtime_bak.sql
```

### 二进制日志 binlog

```sql
-- 查看 binlog 状态
SHOW VARIABLES LIKE '%log_bin%';
SHOW VARIABLES LIKE '%binlog_format%';

-- 查看 binlog 文件列表
SHOW BINARY LOGS;

-- 查看 binlog 内容（命令行）
-- mysqlbinlog binlog.000001

-- 基于时间点恢复
-- mysqlbinlog --start-datetime="2025-06-01 10:00:00" \
--             --stop-datetime="2025-06-01 10:15:00" \
--             binlog.000001 | mysql -u root -p showtime

-- 基于位置点恢复
-- mysqlbinlog --start-position=123 --stop-position=456 \
--             binlog.000001 | mysql -u root -p showtime

-- 重置 binlog
RESET MASTER;
-- 删除指定编号之前的
PURGE MASTER LOGS TO 'binlog.000003';
-- 删除指定时间之前的
PURGE MASTER LOGS BEFORE '2025-01-01 00:00:00';
```

### 主从复制

```
主库 ── binlog ──→ IO线程(从库) ── relay log ──→ SQL线程(从库)
                         ↑                            ↓
                    拉取binlog                  重放到从库
```

```sql
-- 主库配置（my.cnf）
-- server-id = 1
-- log-bin = mysql-bin
-- binlog_format = ROW

-- 从库配置（my.cnf）
-- server-id = 2

-- 主库创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;

-- 查看主库状态
SHOW MASTER STATUS;  -- 记录 File 和 Position

-- 从库配置
CHANGE MASTER TO
    MASTER_HOST = '192.168.1.100',
    MASTER_PORT = 3306,
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'repl_password',
    MASTER_LOG_FILE = 'mysql-bin.000001',
    MASTER_LOG_POS = 123;

-- 启动/停止复制
START SLAVE;
STOP SLAVE;

-- 查看从库状态
SHOW SLAVE STATUS\G
-- 关键字段：
--   Slave_IO_Running: Yes   ← IO线程正常
--   Slave_SQL_Running: Yes  ← SQL线程正常
--   Seconds_Behind_Master: 0 ← 延迟秒数
```

### 读写分离

```
应用层
  ├── 写操作 (INSERT/UPDATE/DELETE) → 主库
  └── 读操作 (SELECT)              → 从库
```

Spring Boot 中通过多数据源配置实现：
- 主数据源：标注 `@Primary`
- 从数据源：配置只读数据源
- 用 AOP 或 AbstractRoutingDataSource 根据方法名/注解自动切换

## 2. ShowTime 业务场景

### 用户权限设计

| 角色 | 权限范围 | 具体权限 |
|------|---------|---------|
| `app_user` | showtime 库 | SELECT, INSERT, UPDATE（应用账号） |
| `report_user` | 特定视图 | SELECT ON v_daily_sales, v_monthly_sales |
| `admin` | showtime 库 | ALL PRIVILEGES（含 DDL） |
| `repl` | 全局 | REPLICATION SLAVE（主从复制用） |

```sql
-- ShowTime 应用账号
CREATE USER 'showtime_app'@'%' IDENTIFIED BY 'strong_password_here';
GRANT SELECT, INSERT, UPDATE ON showtime.* TO 'showtime_app'@'%';
-- 注意：不给 DELETE 权限！所有删除都是软删除（status='deleted'）

-- 报表只读账号
CREATE USER 'showtime_report'@'%' IDENTIFIED BY 'report_password';
GRANT SELECT ON showtime.v_daily_sales TO 'showtime_report'@'%';
GRANT SELECT ON showtime.v_event_occupancy TO 'showtime_report'@'%';
```

### 备份策略

| 策略 | 频率 | 方式 |
|------|------|------|
| 全量备份 | 每天凌晨 3 点 | `mysqldump --single-transaction` |
| binlog 备份 | 持续 | 自动生成，定期清理 7 天前的 |
| 恢复演练 | 每月一次 | 在测试环境恢复备份，验证数据完整 |

### 读写分离时机

演出订票系统的读写特征：
- **读多写少**：浏览演出 99%，下单买票 1%
- **写操作集中**：只有下单、支付、修改用户资料才是写
- **读可以有延迟**：演出列表、座位状态读从库完全没问题（1-2 秒延迟不影响）

```java
// Spring Boot 中通过注解切换数据源
@ReadOnly  // → 走从库
public List<Event> searchEvents(String keyword) { ... }

@Write     // → 走主库  
@Transactional
public Order createOrder(CreateOrderRequest req) { ... }

// 注意：下单后立即查订单 → 必须走主库（主从延迟导致从库还没同步）
@Write
public Order getOrderById(Long orderId) { ... }
```

## 4. 关键提醒

- **--single-transaction**：InnoDB 备份时用这个参数，不加全局锁，业务几乎无感
- **binlog 三种格式**：STATEMENT（记 SQL）、ROW（记行变更）、MIXED（混合）。主从复制用 ROW 格式最安全
- **主从延迟是常态**：业务代码要考虑延迟，下单后立即查订单走主库
- **备份文件要定期验证**：备份不代表能恢复，定期做恢复演练
- **生产上不给应用账号 DDL 权限**：表结构变更走专门的变更流程
- **对于学习来说**：主从复制可以在自己电脑上用 Docker 起两个 MySQL 实例来玩
