# ShowTime — 演出订票系统 · 业务需求文档

> **重要：本文档只描述业务，不涉及任何数据库表设计。**
> 建议你先基于这份文档自己尝试设计表结构，然后再对照参考方案。

---

## 一、项目简介

ShowTime 是一个演出票务平台，用户可以在上面浏览各类演出（演唱会、话剧、体育赛事等），在线选座购票，并查看自己的购票记录。

---

## 二、角色

| 角色 | 描述 |
|------|------|
| **普通用户** | 浏览演出、搜索、选座购票、查看票夹 |
| **管理员** | 管理演出信息、场馆信息、查看销售报表（后台功能，本项目暂不做前端，但数据库需支持） |

## 三、功能需求

### F1. 演出浏览与搜索

用户打开首页可以看到演出列表，每个演出展示：
- 演出名称、海报图 URL
- 演出分类（如：演唱会、话剧、音乐节、体育赛事、相声、展览……）
- 演出场馆名称和所在城市
- 演出状态：即将开售、在售中、已售罄、已结束
- 该演出最低票价和最早场次时间

用户可以：
- 按分类筛选
- 按城市筛选
- 按日期范围筛选
- 按关键词搜索（搜索范围：演出名称、演出简介、场馆名称）
- 按推荐/热度/时间/价格排序

### F2. 演出详情

点击演出卡片进入详情页，展示：
- 演出完整信息：名称、简介/描述、海报、演出时长
- 场馆信息：场馆名、地址、城市
- 所有场次信息：每个场次的开演时间
- 票档与价格：如 VIP 区 ¥1280 / A 区 ¥880 / B 区 ¥580 / C 区 ¥280
- 每个场次每个票档的剩余座位数

### F3. 场馆信息

每个场馆有：
- 场馆名称、城市、详细地址
- 场馆内的固定座位布局：分为不同区域（VIP/A/B/C 等），每个区域包含若干排，每排若干座位
- 不同演出场次可以使用同一场馆，但座位状态独立

### F4. 选座购票（核心功能）

用户选择一个场次后，进入选座页面：

**座位图展示**：
- 以可视化方式展示该场次所有座位
- 用不同颜色区分：灰色=已售 / 红色=已被他人锁定 / 绿色=当前用户已选 / 白色=可选
- 不同票档区域用视觉分隔（VIP 区在前排中间，C 区在后排两侧）
- 展示每个座位的具体编号（如 5排3座）

**选座规则**：
- 用户可以选择 1-6 个座位
- 不能选择已售或已被他人锁定的座位
- 已选的座位有一个锁定时效（如 15 分钟），超时自动释放
- 用户必须在锁定时效内完成支付，否则座位释放

**价格规则**：
- 同一票档所有座位价格相同
- 但可能存在「早鸟票」：开演前 30 天购买享受额外折扣
- 不同场次的同一区域座位价格可以不同（比如周五晚上的比周二下午的贵）

### F5. 下单与支付

用户确认选座后进入下单页面：

**下单流程**：
1. 用户确认所选座位和总价
2. 系统扣减库存（锁定座位 → 标记为已售）
3. 生成订单
4. 生成支付记录
5. 更新座位状态

**业务规则**：
- 如果支付过程中某一步失败，已锁定的座位必须释放，订单状态标为「已取消」
- 支付成功后订单状态变为「已支付」
- 订单超过 15 分钟未支付自动取消，座位释放
- 同一座位在同一场次只能被卖出一次（这是最基本的业务约束）

### F6. 我的票夹

用户登录后可以查看自己的购票记录：

- 订单列表：展示订单号、下单时间、订单状态、总金额、票数
- 点击某个订单可查看详情：
  - 买了哪场演出的什么座位
  - 每张票的详细信息：票档、排、列、价格
  - 订单状态和支付状态
- 支持按订单状态筛选（全部 / 待支付 / 已支付 / 已取消）

### F7. 用户注册与登录

- 手机号 + 密码注册
- 手机号 + 邮箱需唯一
- 昵称（非必填，可修改）
- 登录后才有选座购票功能（未登录只能浏览）

### F8. 管理员功能（给设计留空间）

- 添加/修改演出信息
- 添加/修改场馆信息
- 设置场次和票价
- 查看销售报表（日报/月报）

## 四、业务流程

### 主流程：用户购票

```
进入首页
  → 浏览/搜索演出
  → 点击演出查看详情
  → 选择场次
  → 进入选座页
  → 点击可选座位（绿色标记）
  → 确认选座
  → 进入下单确认页
  → 确认支付
  → 购票成功，生成电子票
```

### 异常流程

- **选座超时**：15 分钟内未支付 → 座位释放 → 订单取消
- **并发冲突**：两人同时选同一座位 → 先锁定的人获得，后者提示「座位已被抢占」
- **支付失败**：订单标记为「已取消」，座位释放
- **库存不足**：场次某区域座位已售罄 → 该区域显示灰色不可选

## 五、业务数据特征

### 数据量级预估

| 数据 | 量级 |
|------|------|
| 场馆 | 几十到几百 |
| 演出分类 | 10-20 个 |
| 演出 | 数百到数千（持续增加） |
| 场次 | 每演出 1-5 场，共数千场 |
| 座位 | 每场次数百到数千个，**总量可达百万级** |
| 用户 | 数十万到百万 |
| 订单 | 随用户增长，百万级 |
| 票 | 和订单成正比 |
| 搜索日志 | 高频写入，千万级 |

### 读写特征

- 读远大于写：浏览演出、搜索、查看详情是主要流量
- 写操作集中：下单瞬间写入密集，且并发冲突可能高（热门演出）
- 日志表只追加不修改不删除

## 六、你能想到的问题（先自己思考）

在开始设计表结构之前，建议你先思考以下问题：

1. 「座位」应该是一张表还是多张表？每个场次要复制一遍座位数据吗？
2. 「锁定」和「已售」是两个状态还是分开放？lock_time 字段放哪里？
3. 订单和票是什么关系？一张订单可以有多张票吗？
4. 一个演出有多个场次，一个场次有多个票档，一个票档对应很多座位——这个层级关系怎么在表中体现？
5. 哪些字段需要建索引？为什么？
6. 哪些表用 InnoDB？哪些可以考虑 MyISAM 或 Memory？
7. 支付记录和订单记录应该分开还是合并？为什么？
8. 搜索日志这种高频写入的表，设计上要注意什么？
9. 用户删除账号后，已有的订单怎么办？（物理删除 vs 逻辑删除）
10. 如果要做「优惠券/促销活动」，表结构要怎么扩展？
