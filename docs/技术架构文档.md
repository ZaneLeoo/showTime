## 三、后端设计

### 3.1 分层架构

```
Controller → Service → Mapper (MyBatis-Plus) → MySQL
```

- **Controller**：接收 HTTP 请求，参数校验，调用 Service
- **Service**：业务逻辑，事务管理
- **Mapper**：继承 MyBatis-Plus BaseMapper，复杂查询用 XML
- **Entity**：实体类，对应数据库表

### 3.2 核心模块

| 模块 | 说明 |
|------|------|
| 用户模块 | 注册、登录、个人信息 |
| 演出模块 | 演出列表、搜索、详情、分类 |
| 场馆模块 | 场馆信息 |
| 选座模块 | 座位查询、锁定、释放 |
| 订单模块 | 下单、支付、票夹 |
| 管理模块 | 演出/场馆管理、销售报表 |

### 3.3 关键配置

MyBatis-Plus 开启：
- 自动填充（createTime / updateTime）
- 逻辑删除（按需）
- 分页插件

## 四、前端设计

### 4.1 UI 方案选择

**推荐方案：Tailwind CSS + 自定义组件**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 不用任何 UI 库 | 完全自由，设计独特 | 造轮子量大，交互细节多 |
| Element Plus | 组件齐全，文档好 | 后台风格明显，C端同质化 |
| Naive UI | 设计感好，TS 友好 | 社区较小，组件不够全 |
| **Tailwind CSS** | 设计自由度极高，无组件风格约束 | 需要 CSS 功底 |
| **Tailwind + Headless UI** | 有交互逻辑，无样式约束 | 组件少，仅基础交互 |

**建议**：采用 **Tailwind CSS + 少量 Headless UI 基座**。理由：
- C端演出票务平台需要独特视觉设计，UI 库的"企业后台感"不合适
- Tailwind 原子化 CSS 让样式定制高效，不产生 dead CSS
- Headless UI（无样式组件）处理 Dialog、Transition、Listbox 等交互，你负责样式
- 一些复杂组件（日历选择、座位图 SVG）本来就需要手写

### 4.2 页面路由

```
/                    → 首页（演出列表）
/search              → 搜索页
/event/:id           → 演出详情
/event/:id/seat      → 选座页（需登录）
/order/confirm       → 下单确认（需登录）
/order/:id           → 订单详情（需登录）
/user/orders         → 我的票夹（需登录）
/user/profile        → 个人信息（需登录）
/login               → 登录
/register            → 注册
```

### 4.3 状态管理（Pinia）

| Store | 职责 |
|-------|------|
| `useAuthStore` | 登录态、token、用户信息 |
| `useSeatStore` | 当前选座状态、已选座位、倒计时 |
| `useOrderStore` | 待支付订单、订单列表 |

## 六、关于索引的说明

当前 `sql/init.sql` 中所有索引均已注释，仅保留 PRIMARY KEY。

这样做的意义：
1. **先有数据**：插入一批测试数据后，查询无索引的全表扫描耗时
2. **逐步加索引**：遇到慢查询时用 EXPLAIN 分析，再加对应的索引
3. **对比效果**：加索引前后 EXPLAIN 输出对比（type 从 ALL → ref/const）
4. **理解代价**：索引加速读但拖慢写，实际写操作时能感受到

后续优化阶段，你会真正理解为什么参考方案建了这些索引。
