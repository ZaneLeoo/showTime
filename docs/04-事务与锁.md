# 04 — 事务 · 锁

> 对应学习文档：第6章 事务、进阶第5章 锁

**这是 ShowTime 项目最核心的知识。选座购票天然就是事务+锁的实战场景。**

---

## 1. 知识清单

### 事务

| 技能 | 操作 |
|------|------|
| 开启事务 | `START TRANSACTION` 或 `BEGIN` |
| 提交 | `COMMIT` |
| 回滚 | `ROLLBACK` |
| 设置保存点 | `SAVEPOINT name` → `ROLLBACK TO name` |
| 查看自动提交 | `SELECT @@autocommit` |
| 设置手动提交 | `SET @@autocommit = 0` |

### 事务四大特性 ACID

| 特性 | 含义 | 在订票系统中的体现 |
|------|------|------------------|
| 原子性 Atomicity | 要么全成功，要么全失败 | 扣库存+建订单+写日志必须一起完成 |
| 一致性 Consistency | 数据从一种正确状态到另一种 | 座位不能超卖，订单金额不能对不上 |
| 隔离性 Isolation | 并发事务互不干扰 | 两个人不能同时买到同一张票 |
| 持久性 Durability | 提交后数据永久保存 | 支付成功的票重启也不丢 |

### 事务隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|-----------|------|
| READ UNCOMMITTED | 会 | 会 | 会 |
| READ COMMITTED | 不会 | 会 | 会 |
| REPEATABLE READ (MySQL 默认) | 不会 | 不会 | 会(部分解决) |
| SERIALIZABLE | 不会 | 不会 | 不会 |

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 锁

| 锁类型 | 粒度 | 适用引擎 | 典型场景 |
|--------|------|---------|---------|
| 全局锁 | 全库 | 所有 | 全库备份 |
| 表锁 | 整张表 | MyISAM/InnoDB | MyISAM 读写 |
| 行锁 | 单行 | InnoDB | 高并发修改不同行 |
| 间隙锁 | 索引间隙 | InnoDB | 防止幻读 |
| 元数据锁 MDL | 表结构 | 所有 | DDL 阻塞 DML |

### 核心语法

```sql
-- 全局锁
FLUSH TABLES WITH READ LOCK;
UNLOCK TABLES;

-- MyISAM 表锁
LOCK TABLES table_name READ/WRITE;
UNLOCK TABLES;

-- InnoDB 行锁（在事务中使用）
SELECT * FROM seats WHERE session_id=1 AND seat_row=5 AND seat_col=10 FOR UPDATE;
-- 跳过已锁定的行
SELECT * FROM seats WHERE session_id=1 AND status='available' FOR UPDATE SKIP LOCKED;

-- 查看锁信息
SHOW ENGINE INNODB STATUS;
```

## 2. ShowTime 核心场景：下单流程

这是整个系统最重要的业务逻辑，一笔下单涉及：

```
BEGIN;
  -- Step 1: 锁定目标座位（行锁）
  SELECT * FROM seats 
  WHERE session_id = ? AND seat_row = ? AND seat_col = ? 
  FOR UPDATE;
  
  -- Step 2: 检查座位状态
  -- (在 Java 代码中判断 status == 'available')
  
  -- Step 3: 更新座位状态
  UPDATE seats SET status = 'locked', lock_time = NOW() 
  WHERE session_id = ? AND seat_row = ? AND seat_col = ?;
  
  -- Step 4: 创建订单
  INSERT INTO orders (user_id, total_amount, status, create_time) 
  VALUES (?, ?, 'pending', NOW());
  
  -- Step 5: 创建票记录
  INSERT INTO order_tickets (order_id, seat_id, price) 
  VALUES (?, ?, ?);
  
  -- Step 6: 创建支付记录
  INSERT INTO payments (order_id, amount, status) 
  VALUES (?, ?, 'pending');
  
COMMIT;
```

**任何一个步骤失败 → ROLLBACK → 座位恢复**

### 并发抢同一座位会发生什么？

```
时间线：
T1: 用户A 事务开始 → SELECT seat_5_10 FOR UPDATE → 获得锁
T2: 用户B 事务开始 → SELECT seat_5_10 FOR UPDATE → ⏳等待...
T3: 用户A 完成下单 → COMMIT → 释放锁，座位状态='sold'
T4: 用户B 获得锁 → 检查 status → 'sold' ≠ 'available' → ROLLBACK
     → 返回给前端：「抱歉，该座位已被抢走」
```

### 为什么会死锁？

```
事务A：先锁 seat_1_5，再试图锁 seat_1_6
事务B：先锁 seat_1_6，再试图锁 seat_1_5

时间线：
T1: A 锁了 seat_1_5
T2: B 锁了 seat_1_6
T3: A 想锁 seat_1_6 → 等 B 释放
T4: B 想锁 seat_1_5 → 等 A 释放
    → 💀 死锁！MySQL 检测到，自动回滚其中一个事务。
```

**解决**：按相同顺序锁定资源（比如永远按 seat_id 升序锁），或者用 `SKIP LOCKED`。

## 3. ShowTime 中的锁策略选择

| 场景 | 策略 | 为什么 |
|------|------|--------|
| 下单锁座 | `SELECT FOR UPDATE`（悲观锁） | 座位是稀缺资源，冲突概率高 |
| 释放过期锁定 | `UPDATE ... WHERE status='locked' AND lock_time < ?`（乐观锁） | 不需要排队等锁 |
| 搜索演出 | 不加锁 | 读多写少，MVCC 足够 |
| 批量分配座位 | `SELECT ... FOR UPDATE SKIP LOCKED` | 跳过已被别人锁定的 |

### 练习 1：转账模拟（理解事务）

-- 经典场景：张三给李四转 1000
-- 建表 account(id, name, balance)

### 练习 2：隔离级别对比（需要两个会话）

```
会话 A：SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
会话 A：BEGIN; UPDATE account SET balance=9999 WHERE name='张三';
会话 B：BEGIN; SELECT * FROM account WHERE name='张三';
        → 能看到 9999 吗？（脏读）

换成 REPEATABLE READ 再试：
会话 A：BEGIN; SELECT * FROM account WHERE name='张三';  -- 1000
会话 B：BEGIN; UPDATE account SET balance=2000 WHERE name='张三'; COMMIT;
会话 A：SELECT * FROM account WHERE name='张三';  -- 还是 1000（可重复读）
```

### 练习 3：复现死锁（需要两个会话）

```sql
-- 准备：seats 表有 id=1 和 id=2 两个座位
-- 会话A：BEGIN; SELECT * FROM seats WHERE id=1 FOR UPDATE;
-- 会话B：BEGIN; SELECT * FROM seats WHERE id=2 FOR UPDATE;
-- 会话A：SELECT * FROM seats WHERE id=2 FOR UPDATE;  -- 等待
-- 会话B：SELECT * FROM seats WHERE id=1 FOR UPDATE;  -- 死锁！
-- 观察：哪个会话被回滚了？SHOW ENGINE INNODB STATUS 看死锁日志
```

### 练习 4：MyISAM 事务对比

## 5. 关键提醒

- **行锁只在事务中生效**：`SELECT FOR UPDATE` 必须在 `BEGIN/START TRANSACTION` 里面
- **行锁依赖索引**：WHERE 条件没命中索引 → 升级为表锁
- **死锁无法 100% 避免**：代码层面要做重试机制
- **REPEATABLE READ** 是 InnoDB 默认隔离级别，通过 MVCC 实现，不是靠锁
- **Spring @Transactional 的坑**：默认只对 RuntimeException 回滚，checked exception 需要 `rollbackFor = Exception.class`
